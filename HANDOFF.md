# Barry — handoff brief (barometer accuracy & experience)

Self-contained task brief for an assistant with no prior conversation context.
Barry is a barometric **pressure-tendency** app: iPhone + Apple Watch app + watch
complication (SwiftUI), backed by a small Python/FastAPI caching proxy. The point
of the app is reading whether weather is coming from the *rate of change* of
pressure, with the phone's barometer filling the gap between hourly airport (METAR)
reports.

## Build / project facts
- The Xcode project is **generated by XcodeGen** from `ios/project.yml`. After
  adding or renaming Swift files, run `cd ios && xcodegen generate`. The
  `.xcodeproj` is gitignored.
- Targets: `Barry` (iOS app), `BarryWatch` (watchOS), `BarryComplication` (widget),
  `BarryTests` (iOS unit tests). Bundle IDs `me.wvr.barry*`, App Group
  `group.me.wvr.barry`, team `46RX35CZGW` (already set in `project.yml`).
- Build the iOS app with the **Barry** scheme. The live barometer features only
  work on a **physical iPhone** (the Simulator has no barometer; the app falls back
  to the METAR value there).
- There is a separate **Python backend** in `backend/` (FastAPI). The app fetches a
  combined JSON payload from it. IMPORTANT: the `reading` object the UI shows
  (trend, confidence, **caveats** like "short_window") is computed **server-side**
  in `backend/app/interpreter.py`. Editing only Swift will not change the live
  message — see Task 1. Backend tests: `cd backend && .venv/bin/pytest`.

## Key files
- `ios/Barry/iOSApp/HeroView.swift` — the glance: current value (live phone reading
  vs METAR), trend chip, verdict, the "Read with care …" honesty note, calibration
  line, micro-trend.
- `ios/Barry/iOSApp/BarometerManager.swift` — phone sensor: `CalibrationModel`
  (multi-point robust offset + `driftPerHour`, persisted to the App Group), motion
  gating, the 60-min in-memory `SampleBuffer`, `microTrend`, `phoneTrace`.
- `ios/Barry/iOSApp/PressureChartView.swift` — main chart; already overlays the
  phone trace (orange) + a divergence band over the last ~60 min, with a 6h/48h
  window toggle.
- `ios/Barry/Shared/Interpreter.swift` — a Swift mirror of `backend/app/interpreter.py`
  (used for an offline fallback). Keep logic in sync with the Python source.
- `ios/Barry/Shared/AppConfig.swift` — `sharedDefaults` (App Group UserDefaults),
  `appGroupID`.
- Backend: `backend/app/interpreter.py`, `backend/app/service.py` (feeds the
  interpreter observed METARs + forecast), `backend/tests/test_interpreter.py`.

## Task 1 — Stop the spurious "Read with care — limited recent data"
**Symptom:** the verdict almost always shows "Read with care — limited recent data,"
and confidence is halved.

**Root cause:** in `backend/app/interpreter.py`, the `short_window` caveat is added
when the trailing slope window spans `< SLOPE_WINDOW_HOURS` (3.0 h). But the newest
sample always lags real-time `now` (last METAR ~20–40 min old; forecast is hourly),
and the series is resampled to a 30-min grid, so the measured window span is almost
always 2–3 h — *just under* 3.0 h — and the caveat trips nearly every time.

**Fix (backend — this is the one that changes the live app):** make the check
reflect genuine coverage instead of demanding a full 3.0 h span. For example, flag
`short_window` only when the available recent history is truly short — e.g.
`(now - chosen[0].t) < 2.0 h` OR fewer than ~4 points fall in the window — rather
than `window_hours < SHORT_WINDOW_HOURS - 1e-6`. Add a tolerance constant if needed.
**Mirror the same change** in `ios/Barry/Shared/Interpreter.swift` (the
`"short_window"` append) to keep parity.

**Acceptance:** a normal hourly-reporting station (≈ last sample 30 min old, points
every ~30–60 min for 3 h) should NOT get `short_window`; a station with < ~2 h of
recent data still should. Update/extend `backend/tests/test_interpreter.py` and run
`pytest`.

## Task 2 — Persist a longer local pressure history
Today `SampleBuffer` in `BarometerManager.swift` is in-memory and capped at 60 min,
so the chart overlay only spans ~1 h and resets on relaunch.

Add a persisted, **downsampled** rolling log of calibrated phone readings (one point
every ~2–5 min, retain ~24–48 h) in the App Group (`AppConfig.sharedDefaults` or a
JSON file in the shared container). Model it as a pure, `Codable`, unit-tested value
type like `CalibrationModel`. Write (throttled) on each trusted sample; load on
launch. This unlocks Task 3.

## Task 3 — "Sensor vs Station" overlay view
New SwiftUI view (e.g. `ios/Barry/iOSApp/SensorComparisonView.swift`) using the
Task 2 history: plot the phone's calibrated continuous trace against the discrete
hourly METAR observations on one axis over several hours, so the divergence and the
phone's finer resolution are both visible. Use `PressureUnit` for display; surface
it below the hero or as a sheet/section in `ContentView.swift`. Re-run
`xcodegen generate` after adding the file.

## Task 4 — Apply the drift you already compute
`CalibrationModel.driftPerHour` is computed but unused. Optionally extrapolate the
offset by the drift slope between METAR calibrations so the live value tracks slow
sensor drift. Keep it conservative (clamp), and keep exposing the value.

## Task 5 — Average at calibration time
`attemptCalibration` uses the single latest raw sample (`buffer.samples.last`). Use a
short trailing average (last ~3–5 min of stationary samples) for the phone pressure
in the offset, to reduce sensor jitter.

## Suggested order
1, then 2 → 3, then 4 and 5. Task 1 is small and immediately visible; Tasks 2–3 are
the bigger experience win (the over-time recorded-vs-reported graph).

## Conventions / gotchas
- Tendency thresholds (`tendency.py` ↔ `Shared/Tendency.swift`) and the interpreter
  (`interpreter.py` ↔ `Shared/Interpreter.swift`) are mirrored across Python and
  Swift **by hand** — change both sides.
- Re-run `xcodegen generate` after adding/renaming files.
- Don't commit per small change; bundle changes (project owner's preference).
